STEP 01

Working safely

Cowork only ever sees the folder you point it at. Inside that boundary, three habits keep you in control: the right permission mode, a review before anything ships, and keeping sensitive data out in the first place.

You are always in control. Cowork does not send your files anywhere unless you use a connector that does, and it asks permission before it touches a new folder. The folder you grant is the boundary. Nothing leaves it until you approve.

Before anything ships, do this

  • Read what Cowork changed. The to-do list shows every step it took.
  • Open the folder and check the output file looks right.
  • Approve it, or redirect and re-run if something is off.

Permission modes

In Settings > Permissions you choose how much Cowork can do without asking:

ModeWhat it doesWhen to use it
Ask every timeCowork checks in before it touches a fileDefault. Start here, always.
Allow within this folderCowork works without asking, inside one folder onlyOnly after a workflow has run correctly at least three times

Sensitive data

The rule is simple: the folder you open is the folder it sees. Do not open a folder that contains customer PII, financial data, passwords, or API keys. When a connector asks for access, grant only what the task needs. If it asks for a whole Google Drive and only needs one folder, say not now and point it at that folder instead.

Verdal's folder is a safe example. It contains brand context and templates only, no customer data. That is why it is safe to use as your practice folder across all six sessions.
STEP 02

Validate a skill before you share it

A skill that works on your machine can still produce off-brand output on someone else's, if their context files look different. Before you hand a skill over, run it once and check it.

Validate before sharing, in order

  • Run it once on a real Verdal example.
  • Check the output is on brand.
  • Confirm it saved to the right folder.
  • Only then, package and share.
Paste into Cowork
validate the weekly brief 
Run the weekly-content-brief skill once on Verdal, show
me the output, and if it looks good, package it so I can
share it with the team.

Pick one, so everyone's output is different:

  • The weekly-content-brief skill
  • A skill you built in Session 3
  • A caption skill

What good looks like

  • Cowork reads the skill definition and pulls brand context before it runs anything.
  • The output lands in projects/, for example weekly-content-brief.md, and it sounds like Verdal.
  • If the output is right, Cowork moves to packaging without you having to ask twice.

Test it

Open the saved brief. If it reads generic or the wrong facts show up, do not package yet. Fix the skill or the context files first, run it again, and only package once the output checks out.

STEP 03

Share with the team

Once a skill is validated, package it into one file. A colleague installs it in one step and gets the exact same output you get, every time.

How it works

Use the Skill Creator panel in the app to package your skill into a single distributable file, for example weekly-content-brief.skill. Hand that file to a teammate any way you like: Slack, email, a shared drive. They drag it into their own Skills panel and it installs, no configuration needed.

share flow 
your skill
  -> package via Skill Creator
  -> weekly-content-brief.skill (one file)
  -> teammate drags it into their Skills panel
  -> same output, same brand, no setup
One thing to agree on first. A packaged skill still needs a folder structure to read from. If your teammate's folder does not have the same layout as yours (context/, templates/, projects/), the skill will still run but may not find the brand files. Agree on a standard layout before you share, the Verdal layout is a good default.

Test it

Ask a teammate to install the packaged skill and run it once on their own copy of the folder. If the output matches what you got, the share worked cleanly.

STEP 04

Check the output

Cowork writes in a confident voice whether or not the fact is true. Before you publish anything, trace every fact, stat, or quote back to the file it should have come from. If it cannot point to one, it made it up.

The rule. Never trust a confident-sounding claim for anything you did not give Cowork the source for. If a draft says "plant food sales are up 40 percent," ask which file says that. If it is not in your files, it made it up.
Paste into Cowork
trace every claim 
Read projects/weekly-content-brief.md. For every
claim or number in it, tell me which file in context/
or data/ it came from. Flag anything you cannot trace
to a file, I will not publish those.

What good looks like

  • Cowork lists each claim next to the file it pulled it from.
  • Anything it cannot trace, it flags instead of defending.
  • You still make the call. Cowork's job is to show its work, not to decide what is safe to publish.

Test it

Pick a draft with at least one stat or quote in it and run the prompt above. If Cowork traces every claim to a real file, it passes. If it cannot name a source for something and still asks you to trust it, that claim does not go out.

When not to hand it to Cowork at all

  • A real, verified human quote or attribution.
  • Anything needing legal or compliance sign-off.
  • Anything where a wrong fact has real cost - pricing, medical-style claims, competitor facts you will publish.
  • Anything involving real customer personal data.
